# master_thesis

Thesis-specific software for:

**Consensus through Learning: Graph Neural Networks for Task Allocation under Motion Constraints**

This directory contains the code developed specifically for the Master's thesis. It extends the BilboLab framework with modules for task assignment, motion planning, trajectory execution, scenario generation, visualization, and reinforcement-learning-based subgoal prediction.

The final thesis evaluation focuses on the **subgoal predictor**. Task assignment is fixed using the Hungarian algorithm in order to isolate the effect of learned intermediate subgoals and wait times. DGNN-GA-related code is included as compatible infrastructure for decentralized assignment, but the final empirical results do not evaluate the full DGNN-GA + subgoal-prediction pipeline.

---

## Scope

This package is research code. It is intended to document and reproduce the simulation pipeline used in the thesis, not to provide a polished standalone framework.

The main evaluated pipeline is:

```text
Hungarian Task Assignment → RL Subgoal Prediction → Motion Planning → Execution
```

The policy does **not** replace the motion planner. It only predicts optional intermediate targets and wait times. Motion between targets is still generated by the classical planning and execution stack.

## Containers

```text
containers/
```

The container layer defines shared data objects for agents, tasks, obstacles, maps, environments, planner outputs, and execution state.

Containers are used as the interface between modules. Assignment, planning, execution, and learning modules read from and write to these objects rather than depending directly on each other.

This keeps the software close to the conceptual pipeline used in the thesis.

---

## General simulation layer

```text
general/
```

Contains the base thesis-specific simulation and agent abstractions. These classes adapt the BilboLab simulation framework to the multi-agent task-assignment and motion-planning setting used in the thesis.

This layer provides the foundation on which the more specialized `universal/` simulation and agent classes are built.

---

## Universal simulation layer

```text
universal/
```

Contains the main integrated simulation pipeline used for thesis experiments.

The universal layer connects:

```text
Task Assignment → Motion Planning → Execution
```

and provides the structure into which the subgoal predictor is inserted.

The intended data flow is:

1. Initialize scenario.
2. Assign tasks.
3. Optionally predict subgoals and wait times.
4. Plan trajectories.
5. Execute trajectories in the simulation loop.
6. Log results.

## Modules

### Task assignment

```text
modules/task_assignment/
```

Contains assignment interfaces and solver implementations.

The final thesis experiments use the **Hungarian algorithm** as the fixed assignment solver. This isolates the learned subgoal predictor from assignment errors.

DGNN-GA-related code is included as a compatible decentralized assignment component. It is relevant for the broader system design and possible deployment setting, but it is **not** part of the final quantitative evaluation.

Use this module as the interface between agent/task state and assigned goals.

---

### Motion planning

```text
modules/motion_planning/
```

Contains the motion-planning pipeline used to generate trajectories between an agent's current state and its current target.

The planner receives:

- agent state,
- target pose,
- map/environment information,
- robot geometry,
- planner configuration.

It returns either:

- a valid trajectory representation, or
- an explicit planning failure state.

The implementation uses an OMPL/FCL-based pipeline and post-processing steps for trajectory generation. The planner remains responsible for collision checking and dynamic feasibility of generated trajectories; the learned policy only changes the intermediate targets passed to this planner.

---

### Execution

```text
modules/execution/
```

Contains the logic that applies planned trajectories inside the BilboLab simulation loop.

The execution module tracks whether an agent is:

- moving along a planned trajectory,
- waiting at a predicted subgoal,
- finished with its task,
- blocked by planning failure,
- ready for replanning.

This layer connects planner output to simulated robot motion.

---

### Subgoal predictor

```text
modules/subgoal_predictor/
```

Contains the reinforcement-learning subgoal prediction code.

The subgoal predictor receives local observations and predicts an action that can include:

- a skip/no-subgoal decision,
- an intermediate subgoal position,
- a wait time.

The decoded action is passed back into the same motion-planning and execution modules used by the baseline pipeline.

Relevant files include:

```text
rl_subgoal_predictor.py
subgoal_manager.py
train_subgoal.py
architecture_notes.md
obs_design_notes.md
reward_design_notes.md
```

The final thesis uses PPO-based training. Older notes or experiments may still mention earlier REINFORCE/bandit-style prototypes; those should be interpreted as development history rather than the final training setup.

---

### Local control

```text
modules/local_control/
```

Contains experiments and utilities related to local control. This module is not the central evaluated contribution of the thesis.

## Scenarios

```text
scenarios/
```

Contains scenario definitions used for debugging, training, and evaluation.

Important scenario types include:

```text
rl_5n_fixed_1gap_2x2.yaml
rl_8n_fixed_1gap_2x3.yaml
rl_10n_fixed_2gap_3x3.yaml
```

These correspond to the main scenario families discussed in the thesis:

- **5n1g**: small single-gap setup used mainly for architecture selection;
- **8n1g**: congested single-gap setup used for the main positive subgoal result;
- **10n2g**: larger two-gap setup used to test multi-gap coordination.

The exact file names may differ across development snapshots. Use the thesis submission tag for the version corresponding to the submitted results.

## Running training

Most thesis code is intended to be run from the `software/` directory.

Example:

```bash
cd software
PYTHONPATH=. python -m master_thesis.modules.subgoal_predictor.train_subgoal
```

Exact training behavior depends on the selected scenario, policy architecture, checkpoint configuration, and training settings inside the script/configuration.

The training pipeline uses the same planner and execution modules as evaluation. This is computationally expensive but avoids training a policy in a simplified environment that differs from the evaluated system.

---

## Running evaluation

Evaluation loads a trained or selected policy checkpoint, runs fixed scenarios, and logs metrics such as:

- termination rate,
- makespan,
- planning failures,
- OMPL wall time,
- gap crossings,
- wait time,
- subgoals reached.

The thesis results are based on fixed evaluation runs, not on visual inspection.

If exact reproduction is required, use the repository state tagged for thesis submission and the checkpoint/configuration corresponding to the relevant experiment.

## Important interpretation notes

The final empirical claim of the thesis is limited.

This codebase contains infrastructure for both task assignment and subgoal prediction, but the thesis evaluation fixes task assignment using the Hungarian algorithm. The main evaluated question is whether a learned subgoal layer can improve execution under motion constraints.

DGNN-GA is therefore included as compatible decentralized assignment infrastructure, not as a fully evaluated component of the final results.

The learned policy does not output low-level controls. It predicts intermediate coordination decisions, while motion execution remains planner-based.
